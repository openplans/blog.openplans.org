b-roll: Paul fights spam from Btwlzyq

Posted on June 8, 2012 by Paul Winkler

“An Internet service cannot be considered truly successful until it has attracted spammers.”

- Rafe's law 

I've been thinking about comment spam a lot lately, and more generally spam in any kind of user-uploaded content.

Here's a story. Back in 2008, we at OpenPlans developed a site for the Orton Family Foundation called communityalmanac.org. It's a nice-looking site for gathering information about the history and character of small towns.

The site got modest traffic, but it still get its share of spam.

I have periodically had the task of cleaning it up. So one day last winter, I found myself with a small pile of spam to clean up and try to figure out how they were getting through. Here's the full tale, pieced together from the server logs.

All this is from one IP address, 110.252.124.249, which is in China. (I later checked this on a few blacklists; no hits. It doesn't appear in our logs earlier or later.)

15:39 Our friend GETs the comments page for Pflugerville, TX, apparently from a search for existing spam; here's the referrer link.

I think the idea here is to assume that prior victims are good targets for further spamming.

15:42 POSTs an anonymous comment. Recaptcha is OK, akismet OK. Score one for spam, zero for our prevention tools.

15:44 POSTs anon comment, recaptcha OK, akismet blocks spam
15:44 POSTs anon comment, recaptcha OK, akismet blocks spam
15:44 POSTs anon comment, recaptcha OK, akismet blocks spam
15:45 POSTs anon comment, recaptcha OK, akismet blocks spam
15:45 POSTs anon comment, recaptcha OK, akismet blocks spam
15:46 POSTs anon comment, recaptcha OK, akismet blocks spam

So far I'm suspecting a bot that has solved recaptcha.

Now we're doing a better job of filtering and our friend gives up on comments… but doesn't go way:
15:46 GETs the home page
15:46 GETs the Shickshinny-PA page
15:46 Attempts to POST a page, gets redirected to login.
15:47 Registers successfully (recaptcha OK), creates account 'btwlzyq'

Oh, hello there btwlzyq. Looks like you're human after all.

From now on we have no protection against this user from either recaptcha or akismet.

15:48 POSTs a page
15:48 POSTs a page
15:48 POSTs a page
15:49 POSTs a page
15:51 POSTs a page
15:51 POSTs a page

The spammer apparently now realizes that there's nothing on these pages.

15:52 creates text on a page.
15:53 creates text on a page.
… etc.

Btwlzyq wandered away at 16:13, apparently never to return. All together, he/she/it made 67 POSTs, about 18 of which successfully added spam to Stories.

Lessons

No one antispam measure is a silver bullet. Some defenses are effective only against bots, some against humans as well. A combination of defenses is better than one.

Don't leave the door wide open. We were using Akismet for comments, but not for “story” content on pages. That's been fixed.

Finally, check that you're implementing each of your defenses correctly. I noticed that we weren't submitting the IP address or user agent to Akismet, reducing its effectiveness.

Logs matter. I had added some logging earlier, which is how I knew when akismet and recaptcha were working and when not. But I had to piece together what happened by cross-referencing the access and event logs, with only timestamps to connect them, and the messages were not consistently formatted. It would have been impossible with a lot of traffic.

Honeypot fields

One defense I added after this incident is a honeypot field on the comment form. This won't do anything about human spammers like Btwlzyq and friends, but it has cut down on bot spam.

I had heard of the technique but never tried it. It's dead easy to implement – just add a hidden form field which is expected to be empty. If it's filled out, we assume that the submitter is a bot, mindlessly filling out all available form fields; so we block the submission.

A more sophisticated implementation would randomize all field names, so a bot can't be programmed to know which is which.

Since adding the honeypot, logs show it has caught about 20% of the comments.

Do it again, and again

There are a lot more things we could do in the future. Throttling; IP blacklisting; more content filter services (blogspam, mollom, defensio, et al); link counting.

But spam is a problem on every site we build that allows any user input. I don't want to have to build all that ad-hoc in a reactive fashion every time we launch a new site.

To that end I'm working on some software, built on some good prior art, that should be straightforward to plug in to any python web framework, and allow adjusting your defenses via configuration, and allow plugging in new filters without further touching your app's code.

I'll be talking about all this at PyGotham II this Friday, June 8.
